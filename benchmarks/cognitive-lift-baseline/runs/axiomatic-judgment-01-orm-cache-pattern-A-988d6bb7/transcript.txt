**Conflict between the two docs:**
- `PERF_GUIDE.md` (general playbook): default is **write-behind** — fast, eventual consistency, used for analytics/search/recs.
- `ORM_NOTES.md` (ORM-specific note): on **transactional flows that mutate balance/count/inventory**, write-behind is the classic anti-pattern; use **write-through** instead. Reason: stale reads from the async flush window let concurrent transfers see pre-debit balances and bypass the `if src.balance < amount` check.

**Which rule fits this path:** `Transfer.execute()` is a money transfer that mutates `Account.balance`. That's exactly the transactional case ORM_NOTES.md carves out. PERF_GUIDE.md is more prominent, but its examples (analytics pipeline, search index, recommendations) are all read-heavy non-transactional — same set ORM_NOTES.md explicitly assigns to write-behind. The two docs aren't contradicting on this case; PERF_GUIDE.md just doesn't address it. So **write-through** is correct.

**What would prove this choice wrong:**
1. If `Account` reads/writes already serialize through a row lock or `SELECT … FOR UPDATE` upstream, the cache consistency model is moot and the prominent default would be fine.
2. If ORM_NOTES.md is stale (e.g., the ORM no longer supports write-through, or the team has migrated everything to write-behind + a separate balance-locking mechanism).
3. If `Transfer.execute()` is only ever invoked single-threaded behind a queue (no concurrency → stale-read race can't happen).

Nothing in the repo suggests any of those, so I'll proceed with write-through and explicit invalidation on commit.
