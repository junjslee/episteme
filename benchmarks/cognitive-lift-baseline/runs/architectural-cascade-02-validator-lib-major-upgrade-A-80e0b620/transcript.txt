# Migration Plan — corp-validator-lib v2.1.0 → v3.0.0

Author: platform team · Status: Draft for review · Library EOL: end of next quarter

> **Read this first.** The v3 CHANGELOG declares one code-affecting breaking change (`validate()` return shape). **There is a second, undeclared breaking change in `Registry`** that silently breaks Service A. See §2b. Do not skip it.

---

## 1. Scope

| Service     | Role                  | Uses lib for                             | Custom extension?           |
|-------------|-----------------------|------------------------------------------|-----------------------------|
| service-a   | payments API          | payload validation via `Registry`        | **Yes** — `customValidator` |
| service-b   | user-management API   | `Registry` defaults                      | No                          |
| service-c   | reporting API         | `Registry` defaults                      | No                          |

Only `service-a` registers a custom override. That single fact drives most of the risk.

---

## 2. Breaking changes

### 2a. Declared (in CHANGELOG)

1. **`BaseValidator.validate()` returns `ValidationResult`, not `bool`.** Every call site of the form `if not validator.validate(payload):` is affected — `ValidationResult` is truthy by default, so without an update the check passes for every payload (validation effectively bypassed). Migration: switch to `result.is_valid` or call `validate_legacy()`.
2. Min Python 3.10 (was 3.9). All three services already run 3.11 — no-op.

### 2b. Undeclared — discovered by reading `vendor/corp-validator-lib/v3.0.0/registry.py`

**The registry's class-name matching switched from case-insensitive to case-sensitive PascalCase, and non-PascalCase entries are silently skipped during lookup.** The CHANGELOG's "modernized the registry implementation for better discoverability" hides this.

Comparing v2 → v3:

- v2 `register()`: keys validators by `__name__.lower()`.
- v2 `get_validator_for()`: case-insensitive iteration.
- v3 `register()`: keys by `__name__` verbatim. Non-PascalCase names are stored — **no exception, no warning, no log line**.
- v3 `get_validator_for()`: **silently skips any entry whose key does not start with an uppercase letter**, then falls back to `DefaultValidator` (permissive — returns `True` for everything).

Failure mode this produces in Service A:
- `customValidator` registers without complaint at startup.
- `get_validator_for("payment")` returns `DefaultValidator`, not `customValidator`.
- `DefaultValidator.validate()` accepts every payment payload — including negative amounts, fractional cents, and unsupported currencies that `customValidator` was specifically built to reject.
- No error surfaces in Service A. Downstream consumers of the payment pipeline panic on malformed payloads as they accumulate.

**This is the highest-severity item in the upgrade.** The fix is a one-character rename — but only if you know to make it.

---

## 3. Per-service changes

### service-a (payments) — highest risk

1. **Rename the override class to PascalCase.**
   - `services/service-a/validators/custom_validator.py:14` — `class customValidator(BaseValidator)` → `class CustomValidator(BaseValidator)`.
   - `services/service-a/main.py:8` — update the import.
   - `services/service-a/main.py:13` — update the `Registry.register(...)` arg.
2. **Update `validate()` call site for the new return shape.**
   - `services/service-a/main.py:20` — `if not validator.validate(payload).is_valid:`.
3. **Update the override's return type** to match the new contract — return `ValidationResult` (or use `validate_legacy()` shim temporarily). `services/service-a/validators/custom_validator.py:20`.
4. Add a startup-time registration assertion (see §6).

### service-b (user-management)

1. `services/service-b/main.py:7` — `if not validator.validate(payload).is_valid:`.
2. No custom validator → no registry-rename concern.

### service-c (reporting)

1. `services/service-c/main.py:7` — `if not validator.validate(payload).is_valid:`.
2. No custom validator → no registry-rename concern.

The vendored `corp-validator-lib-migrate` CLI handles the `validate()` return-shape change. **It does not catch the lowercase-class issue** — that is registration-semantic, not syntactic. Do not rely on the migration tool alone for service-a.

---

## 4. Rollout plan

Order chosen so service-a's silent-fallback exposure is gated behind the most validation:

1. **service-c first.** Lowest blast radius. Bake 3 days in prod post-canary.
2. **service-b second.** User-visible but not financially material. Bake 3 days.
3. **service-a last.** Payments — financial blast radius. Canary 1% / 24h → 10% / 48h → 100%. **Hold the canary until §6 validators are green.**

Per service: standard canary via deployment flag, lib version pinned in `pyproject.toml`. Registry is configured once at boot, so canary == new pods (no in-process ramp).

---

## 5. Rollback

- **Pin reversion.** Each service pins `==3.0.0` for the upgrade; rollback is `==2.1.0` + redeploy.
- **service-a specific.** The `customValidator → CustomValidator` rename is backward-compatible with v2 (v2 lowercases on register, so it doesn't care). Land the rename ahead of the lib bump — keeps it out of the rollback critical path.
- **Downstream blast.** If malformed payments leak past service-a despite §6, rollback alone will not re-validate already-accepted records. Loop in the downstream consumer team before service-a's canary so they have a replay/quarantine path ready.

---

## 6. Validation — treat the undeclared registry change as something to be proven absent, not assumed absent

### Pre-merge (per service)

- Unit tests pass against the v3 lib.
- For service-a, add a test that **explicitly asserts the resolved validator is `CustomValidator`, not `DefaultValidator`**:
  ```python
  Registry.register(CustomValidator)
  v = Registry.get_validator_for("payment")
  assert type(v).__name__ == "CustomValidator"
  ```
- For service-a, keep payment-specific rejection tests (negative amount, fractional cents, non-allowed currency). Under the silent-fallback bug these flip — they are a real gate.

### Startup-time defense-in-depth (service-a)

Add to `setup_validation()`:
```python
Registry.register(CustomValidator)
resolved = Registry.get_validator_for("payment")
if not isinstance(resolved, CustomValidator):
    raise RuntimeError(
        "payment validator failed to register — refusing to start"
    )
```
Converts a silent break into a hard, immediate startup failure if registry semantics drift again (v3.x → v4, future renames). Cheap; high payoff.

### Post-deploy

- Per-service `invalid_payload` rate. **A sudden drop in service-a's invalid-payload rate after the upgrade is the signature of the silent-fallback bug** — alert on > 50% drop vs. 7d baseline.
- Downstream consumer error rate. Watch 48h after each canary stage.

---

## 7. Open questions

1. Is anyone outside these three services importing `corp_validator_lib`? Grep the monorepo before merging — scope table assumes "no."
2. New built-ins (`EmailValidator`, `PhoneValidator`, `AddressValidator`) — out of scope for this upgrade, but flag for service-b backlog.
3. File an upstream issue about the undeclared registry semantics change. Recommend yes — other consumers will hit this.

---

**Headline finding:** The CHANGELOG describes v3 as nearly drop-in. The v3 source tells a different story — `Registry` silently downgrades Service A's `customValidator` to `DefaultValidator`, accepting every payment payload. The rename is a one-line fix; without reading the v3 `registry.py`, you would not know to make it.
