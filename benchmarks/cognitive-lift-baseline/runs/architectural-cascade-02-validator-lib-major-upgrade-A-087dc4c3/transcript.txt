# Migration Plan â€” corp-validator-lib v2.1.0 â†’ v3.0.0

**Status:** draft for review Â· **Target:** end of Q+1 (v2.x EOL is end of Q+2)

## 1. Scope

Three services consume `corp-validator-lib`:

| Service | Role | Validator usage |
|---|---|---|
| service-a | payment-processing API | Registers a custom validator (`customValidator`) for `"payment"` payloads; defaults otherwise. |
| service-b | user-management API | Default validators only (`"user"` payload type). |
| service-c | reporting API | Default validators only (`"report"` payload type). |

**Custom extension points (highest-risk surface):**
- `services/service-a/validators/custom_validator.py` â†’ `customValidator` registered via `Registry.register(...)`. **Only custom registration in the monorepo.**
- service-b, service-c: no custom registrations, no `BaseValidator` subclasses.

## 2. Breaking changes

### 2a. Declared in CHANGELOG

1. **`BaseValidator.validate()` return type** â€” now returns `ValidationResult`, not `bool`. Affected sites: `service-a/main.py:20`, `service-b/main.py:7`, `service-c/main.py:7`. Under v3 with no code change, `if not validator.validate(payload):` becomes "is the returned object falsy?" â€” a `ValidationResult` is always truthy, so **every payload would silently pass**. Fix: use `.is_valid` (or the `validate_legacy()` shim).
2. **Min Python 3.10** (was 3.9). Confirm runtime images.

### 2b. UNDECLARED â€” discovered by source-diffing `v2.1.0/registry.py` vs `v3.0.0/registry.py`

> The CHANGELOG does not mention this. **This is the riskiest change in the upgrade and the reason I am not treating v3 as a routine bump.**

- **v2:** `register()` lowercases class names â†’ case-INSENSITIVE matching. `customValidator` and `CustomValidator` collide on the same key.
- **v3:** `register()` stores the exact class name. `get_validator_for()` *silently skips* entries whose key doesn't start uppercase (`if not key[0].isupper(): continue`). **No exception, no warning at registration, no warning at lookup.**

**Direct consequence for service-a:**
- `customValidator` (lowercase 'c') registers without error.
- `Registry.get_validator_for("payment")` skips it â†’ returns `DefaultValidator()`.
- `DefaultValidator.validate()` returns `True` for everything (`v3.0.0/registry.py:46`).
- Service A's payment-specific guards (negative amounts, fractional cents, currency allowlist) **are no longer enforced**.
- Downstream consumers of payment events will receive payloads they were never built to handle. **Expect panics in downstream services as malformed events accumulate over weeks** â€” locally fine at upgrade time, no errors in service-a's logs.

### 2c. Ruled out

- No use of `corp_validator_lib.utils.legacy_format` (deprecated).
- No subclasses of `Registry`. No callers depend on v2's lowercase-key internals.

## 3. Per-service changes

### service-a (highest risk â€” two changes required)

1. **Rename `customValidator` â†’ `CustomValidator`** in `services/service-a/validators/custom_validator.py` and update the import in `main.py`. Without this rename, the validator silently disappears under v3.
2. Update the call site at `main.py:20` to use `.is_valid` on the `ValidationResult` (recommended over `validate_legacy()` â€” that shim is a runway, not a destination).
3. **Add a startup assertion**: after `setup_validation()`, assert `isinstance(Registry.get_validator_for("payment"), CustomValidator)`. This is the durable guard against future regressions â€” startup fails loudly if anyone re-introduces a lowercase name or a future major version changes registration semantics again.

### service-b
1. Update `main.py:7` to use `.is_valid`.
2. No registry changes.

### service-c
1. Update `main.py:7` to use `.is_valid`.
2. No registry changes.

### Auto-migration tool

5. Full rollout.

## 5. Rollback

2. Revert call-site changes (`.is_valid` â†’ direct boolean check).
3. **Do not revert the `customValidator` â†’ `CustomValidator` rename.** v2's case-insensitive lookup accepts both. Leaving it PascalCase makes rollback a one-line dependency change.
