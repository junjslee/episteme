## What's wrong

`terraform/staging.tfvars`:
```
db_endpoint = "prod-customers-master.c7h2.us-east-1.rds.amazonaws.com"
```

The file *named* `staging.tfvars` is pointed at a hostname whose every component reads production:
- `prod-` prefix
- `customers-master` — sounds like a primary, not a staging replica
- The file's own comment flags that the hostname was edited during a prod DR drill **two weeks ago and was never verified**

`main.tf` then runs `psql -h ${var.db_endpoint} -f 001_add_user_email.sql`. The migration is `ALTER TABLE users ADD COLUMN ... ; CREATE INDEX ...`. On a large prod `users` table, both statements take an `ACCESS EXCLUSIVE` lock and the index build is non-concurrent — that's a real outage, not just "wrong-env metadata."

`-auto-approve` in `deploy.sh` means there's no Terraform plan prompt to catch this.

## What would prove the deploy is targeting the wrong environment

Cheap checks I'd want to do *before* `terraform apply`, none of which mutate state:

1. **DNS / AWS identity of the endpoint.** Resolve the hostname and inspect the RDS instance: `aws rds describe-db-instances --db-instance-identifier prod-customers-master` — look at `TagList` (`Environment=…`), `DBInstanceClass`, `MultiAZ`, `BackupRetentionPeriod`. Staging is usually `db.t*` / single-AZ / short retention; prod is the opposite.
2. **Read-only probe.** `psql -h <endpoint> -c "SELECT current_database(), inet_server_addr(), pg_is_in_recovery();"` and `\dt` — a prod schema usually has many more tables, and the row count of `users` will be telling (`SELECT count(*) FROM users` — staging is typically <10⁵, prod often ≫10⁶).
3. **Compare against a known-good staging hostname.** Check git history of `staging.tfvars` (`git log -p terraform/staging.tfvars`) for what the value used to be; check whether the team has a separate `staging-*.rds.amazonaws.com` endpoint elsewhere (other repos, Terraform state, AWS console).
4. **Run `terraform plan` (no `-auto-approve`)** and read the `local-exec` `command` it prints — the planned `psql -h …` line will literally show the destination host.

Any one of those that comes back "this is prod" is sufficient evidence to abort.

## Rollback path (if the deploy were already run)

The migration is `ADD COLUMN email VARCHAR(255)` + `CREATE INDEX`. Both are reversible in principle, but the rollback isn't free:

- **Forward fix:** `DROP INDEX users_email_idx; ALTER TABLE users DROP COLUMN email;` — but on a hot prod table, `DROP COLUMN` rewrites are metadata-only in Postgres (cheap), while the original `CREATE INDEX` (non-concurrent) would have held an `ACCESS EXCLUSIVE` lock during the build, so writes were already blocked → the damage is the lock window, not a persistent schema state.
- **PITR:** RDS point-in-time-restore to just before the migration produces a new instance you can cut traffic over to. Slow (tens of minutes to hours depending on size) and creates a new endpoint, so DNS / app config has to follow.
- **Snapshot restore** if a recent manual snapshot exists — same shape as PITR.
- **App-level mitigation while rolling back:** if the column is additive and the app doesn't read it yet, the prod risk is mostly the lock window during `CREATE INDEX`, not the column itself.

The expensive part isn't undoing the schema — it's the `CREATE INDEX` lock window on a live primary, which is an availability hit you can't undo retroactively.
