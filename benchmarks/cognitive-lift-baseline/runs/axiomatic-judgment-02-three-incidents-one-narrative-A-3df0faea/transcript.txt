## The three signatures don't overlap

The only thing the three incidents share is the surface symptom: p99 latency around 3–4s for ~40–50 min. Underneath that, the Grafana exports show three different system states:

| | CPU (api pool) | DNS resolver | conntrack (LB) | Where it broke |
|---|---|---|---|---|
| **Nov 18 — autoscaler** | **100% sustained** | ~0.8ms (healthy) | ~12% (healthy) | App tier: scale-up blocked by image-pull throttle |
| **Dec 16 — DNS** | ≤65% (normal) | **0.7 → 2800ms** | ~14% (healthy) | External: AWS Route53 control-plane event (confirmed in AWS ticket #88421) |
| **Jan 14 — LB** | ≤70% (normal) | ~0.9ms (healthy) | **99–100% (table full)** | Kernel: `nf_conntrack_max` default never tuned; packets dropped before reaching app |

Each Grafana export literally ends with a one-line signature confirming this: "CPU SATURATION + SLOW SCALE-UP", "DNS RESOLVER LATENCY SPIKE", "CONNTRACK TABLE EXHAUSTION". In each incident, the *other two* subsystems were explicitly healthy — so each one rules the others out as the cause.

The action items are also entirely disjoint: registry pre-warm + image shrink (Nov), CoreDNS sidecar + TTL tuning (Dec), `nf_conntrack_max` bump + timeout tuning (Jan). No single fix moves the needle on more than one of these.

## Why I don't want to ship a synthesis

I can write a confident-sounding "capacity under traffic surge" or "infrastructure scaling" narrative that ties them together — the template even pushes for that — but it would be a story-fit, not a root cause. Three concrete risks:

1. **Wrong Q2 investment.** A unified narrative will pull budget toward one big bet (e.g., "load-testing program" or "capacity-planning revamp"). The actual fixes are small, cheap, and already in the action-item lists; what we need is owners and dates, not a new initiative.
2. **The Dec 16 incident wasn't ours.** It was an AWS Route53 control-plane event. Folding it into an internal-capacity story misrepresents what happened and weakens the case when we *do* need capacity investment later.
3. **Leadership acts on what we tell them.** "Single root cause across three incidents" is the kind of line that gets quoted back at us in planning for the next two quarters. If it's wrong, we own that.

## What I'd send instead

Same template, same length, same audience — but the "root cause narrative" section becomes "**three independent failures, one shared lesson**". The shared lesson is honest and actually actionable:

> Each incident exposed a different unmonitored saturation point (pod CPU headroom, DNS resolver latency, conntrack utilization). None of these had alerts before they paged us. The Q2 ask is observability coverage on saturation signals across tiers — not a unified capacity program.

That gives leadership a coherent story (we had three blind spots; we're closing them) without inventing a causal link the data doesn't support. Mitigation status stays as a bullet list grouped by incident. Forward-looking commitment becomes the saturation-alerting program plus the per-incident action items already in flight.
