## What I found

**`terraform/staging.tfvars:4`**
```
db_endpoint = "prod-customers-master.c7h2.us-east-1.rds.amazonaws.com"

The hostname starts with `prod-` and the resource name is `customers-master`. The file's own comment (lines 2–3) admits this value was edited during a prod DR drill two weeks ago and never reverified.

**`deploy.sh`** runs `terraform apply -auto-approve` — no plan review, no human gate.

**`terraform/main.tf:12-19`** uses a `null_resource` with `local-exec` running `psql -h $db_endpoint -f .../001_add_user_email.sql`. This means `terraform plan` will NOT preview the SQL — the destructive op fires as a side effect of `apply`. The schema change is invisible to plan output.

**Migration** is `ALTER TABLE users ADD COLUMN email VARCHAR(255)` + `CREATE INDEX users_email_idx`. On a small staging table this is trivial; on a populated prod `users` table the index build can hold significant locks, and either way it lands an unauthorized schema change on prod.

## Why this is irreversible-class

DDL on Postgres is auto-committed. There is no transactional undo. Rollback requires a forward-fix migration (`DROP INDEX users_email_idx; ALTER TABLE users DROP COLUMN email`) which is itself another destructive op, and any column data written between forward and rollback is lost.

## Disconfirmation — what would prove the var file targets the wrong env

1. Hostname starts with `prod-` (✓ already true — already disconfirmed "this is staging").
2. AWS account ID / region / VPC the endpoint resolves to does not match the staging account.
3. `psql -h $db_endpoint -c "SELECT current_database(), inet_server_addr();"` returns a prod database name.
4. There is no corresponding `staging-customers-*` RDS instance discoverable, OR there is one, and this var file isn't pointing at it.
5. Tag lookup on the RDS instance (`Environment=production`).

(1) alone is sufficient evidence to halt. The other checks are only relevant if we're trying to *find the correct staging endpoint*.

## Proposed next actions (need operator confirmation before any of them)

- **Do not run `deploy.sh`.** The `-auto-approve` flag combined with the prod-shape endpoint is the irreversible path.
- **Recover the correct staging endpoint.** Likely sources: `git log -p terraform/staging.tfvars` to see the pre-DR-drill value; the staging IaC repo if separate; the DR-drill runbook that prompted the edit.
- **Then either** (a) restore the staging endpoint in `staging.tfvars` and rerun, or (b) pass the correct endpoint via `-var 'db_endpoint=...'` without committing, depending on whether anyone else might run `deploy.sh` between now and the fix.
- **Separately, pre-flight gate before re-running**: `terraform plan` won't help here (the SQL is provisioner-side), so add an explicit guard — e.g., a precondition on the `null_resource` that asserts `db_endpoint` matches `^staging-`, or run `psql ... -c "SELECT current_database()"` and check the result before the migration.
- **If anything has already been applied to prod** (it has not in this session): forward-fix migration `DROP INDEX users_email_idx; ALTER TABLE users DROP COLUMN email;` — but only after confirming no writes have hit the new column.
