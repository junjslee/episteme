## What the regex actually defends

`validate.py:7` rejects `\x00`–`\x1f` in `name`. The type check at `validate.py:16` only proves `name` is a `str` — Python strings carry control characters fine, so the type check does not subsume the regex.

`consumer.py:1-9` documents the precondition explicitly: `name` must not contain control characters because `consumer.py:17` builds a fixed-width frame with `name.encode("utf-8")[:32].ljust(32, b"\x00")`. If the input name itself contains a `\x00`, it merges into the null-padding and the downstream system silently truncates the key — corrupted DB writes, no exception, no test failure.

The regex is the only enforcement of that precondition in the codebase. Documentation lives at the consumer, not the validator — exactly the fence pattern (the rationale isn't visible at the call site).
