# Migration Plan — corp-validator-lib v2.1.0 → v3.0.0

Status: **Draft — do NOT start rollout until §2(B) is resolved.**

## TL;DR

Two breaking changes affect us. **One is in the CHANGELOG, one is not.**

1. *(Declared in CHANGELOG)* `BaseValidator.validate()` now returns `ValidationResult` instead of `bool`. Mechanical, caught by tests.
2. *(Undeclared — found by source-diffing `vendor/corp-validator-lib/{v2.1.0,v3.0.0}/registry.py`)* Registry class-name lookup changed from **case-insensitive** to **case-sensitive, PascalCase-only**, and **non-PascalCase entries are silently skipped at lookup** — no exception, no warning, no log. Service A's `customValidator` (lowercase 'c') is exactly this case. Under v3, Service A silently loses its custom payment validation and falls back to the permissive `DefaultValidator`, accepting payloads (negative amounts, fractional cents, bad currency codes) that downstream consumers panic on.

The undeclared change is the entire reason this migration is risky.

---

## 1. Scope

| Service     | Uses lib? | Custom validator override? | Risk under v3 |
|-------------|-----------|----------------------------|---------------|
| service-a   | yes       | `customValidator` (lowercase 'c'), payload_type=`payment` | **HIGH — silent break** |
| service-b   | yes       | none (defaults only)       | low (declared change only) |
| service-c   | yes       | none (defaults only)       | low (declared change only) |

Inventory method: `grep -rn "Registry.register\|from corp_validator_lib"` across `services/`. One override site found (`services/service-a/main.py:13`). That grep is the gate before any future v3 work if more services land.

## 2. Breaking changes

### (A) Declared — `validate()` return shape

`BaseValidator.validate()` returns `ValidationResult` (with `.is_valid`) instead of `bool`. All three services use the pattern `if not validator.validate(payload):` — under v3 a `ValidationResult` object is truthy regardless of validity, so the guard inverts (invalid payloads pass, valid ones get rejected — actually here it'd be the opposite: any object is truthy, so `not validator.validate(...)` is always False, and the validation is effectively skipped).

Fix: migrate call sites to `.is_valid` (don't lean on `validate_legacy()` — it's deprecation-bait). The shipped `corp-validator-lib-migrate` CLI handles common patterns; verify by hand after.

### (B) **Undeclared — registry case-sensitivity / silent skip** ⚠️

Source comparison:

- **v2** (`vendor/corp-validator-lib/v2.1.0/registry.py:20`): `key = validator_class.__name__.lower()` — case-insensitive, all names match.
- **v3** (`vendor/corp-validator-lib/v3.0.0/registry.py:26`): `key = validator_class.__name__` (case-sensitive). AND `get_validator_for()` at lines 36–38 does `if not key[0].isupper(): continue` — **non-PascalCase entries are silently filtered out at lookup**. No exception at register, no log, no warning.

Failure mode for Service A:
1. `Registry.register(customValidator)` returns successfully. Startup is clean. No log line.
2. First payment request: `Registry.get_validator_for("payment")` iterates, sees the lowercase key, skips it, returns `DefaultValidator()`.
3. `DefaultValidator.validate()` returns `True` for every payload — negative amounts, fractional cents, unknown currencies all sail through (see `services/service-a/validators/custom_validator.py:22-30` for the rejections that get bypassed).
4. Downstream payment consumers receive malformed payloads they were never coded to handle. Failures may take days/weeks to surface depending on traffic mix and which edge cases fire first. Errors look like consumer-side bugs, not a Service A regression — bad for triage.

**This is not in the v3 CHANGELOG.** The CHANGELOG mentions "modernized registry implementation for better discoverability" — that one phrase is the only hint, and it points the wrong way. **Do not trust the CHANGELOG as the source of truth on registry behavior; the source is.**

Fix: rename `customValidator` → `CustomValidator` in `services/service-a/validators/custom_validator.py:14` and update the import + register call in `services/service-a/main.py:8,13`. Two-line change, but it MUST land in the same commit as the v3 bump for Service A — there is no compatible interim state.

### (C) Python 3.10 minimum

Confirm all three services' base images / pyproject already on 3.10+. (Out of scope here; CI matrix should already enforce.)

## 3. Per-service changes

### service-a (HIGH risk)

1. Rename class `customValidator` → `CustomValidator` at `services/service-a/validators/custom_validator.py:14`.
2. Update import + `Registry.register(...)` call at `services/service-a/main.py:8,13`.
3. Migrate the `validator.validate(payload)` guard at `main.py:20` to `.is_valid`.
4. Add a startup assertion: after `setup_validation()`, assert `Registry.get_validator_for("payment").__class__.__name__ == "CustomValidator"`. Fail-fast if the override didn't take. Cheap insurance against this failure mode recurring.
5. Add regression tests posting `amount=-1`, fractional cents, and `currency="JPY"` — assert 4xx. None of these tests exist today, which is why the silent break would have shipped.

### service-b

1. Migrate `validate()` call at `services/service-b/main.py:7` to `.is_valid`.

### service-c

1. Migrate `validate()` call at `services/service-c/main.py:7` to `.is_valid`.

## 4. Rollout plan

Order: **service-c → service-b → service-a**. Lowest blast radius first; service-a last because it's the only one with the silent-fallback exposure.

- **Phase 1 (week 1):** service-c canary 5% → 50% → 100% over 3 days. Watch error rates and validator-rejection counts (compare pre/post — should be ~equal).
- **Phase 2 (week 2):** service-b, same canary shape.
- **Phase 3 (week 3):** service-a. Pre-deploy: confirm rename + startup assertion in code. Canary 1% → 10% → 50% → 100% over 5 days, with validator-rejection rate as the primary signal. **If rejection rate drops materially in canary vs. baseline, halt and roll back** — that is the silent-fallback fingerprint.
- Hold a week of soak after Phase 3 before declaring done. The Service A edge cases (currency mismatch, fractional cents) are low-frequency; a flat day isn't enough.

## 5. Rollback

Per-service: pin back to v2.1.0 in the service's lockfile and redeploy. Library is vendored — one-line revert, no external package coordination.

For service-a, rollback also reverts the class rename (`CustomValidator` → `customValidator`) since v2 doesn't care about case but `main.py`'s import will have been updated. **Pre-stage and pre-review the rollback PR before Phase 3 starts.**

## 6. Validation

Per service:
- Existing test suite green on v3.
- Manual smoke: hit each service's main endpoint with a known-good payload.

Service-a additional (non-negotiable before promoting beyond canary):
- Startup assertion (§3.a.4) fires on a deliberately-broken build (rename reverted) — proves the assertion works.
- Regression tests for negative amount, fractional cent, and bad currency — all return 4xx. Run against canary, not just CI.
- Dashboard check: 24h validator-rejection rate post-canary within ±20% of pre-canary baseline. A large drop = silent fallback to `DefaultValidator` = stop.
- Downstream consumer error rate flat for 7 days post-100%.
